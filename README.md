# Invoca Docs

Mintlify-powered documentation site for Invoca's Support and Developer Portal.

Production: https://docs.invoca.com (deployed automatically on push to `main`).

## Local development

```bash
npm i -g mint
mint dev
```

Runs a local preview at http://localhost:3000.

## Repo layout

```
docs.json              # Mintlify config: navigation, theming, versions
index.mdx              # Site landing page
en/latest/             # Developer Portal content (current API version)
  api_documentation/
  basics/
  web_integration/
s/article/             # Invoca Support articles
images/                # Shared image assets
logo/                  # Brand logos
```

## Deployment

The Mintlify GitHub App watches `main`:

- **Push to `main`** → auto-deploys to production.
- **Open a PR** → preview deployment URL is posted on the PR.

No CI build step is required — Mintlify builds from the repo on its side.

## Versioning

The Developer Portal supports multiple API versions via a version switcher in the nav bar. Versions are configured in `docs.json` under `navigation.products[].versions`.

Today there is one version: `latest`. Its content lives under `en/latest/`.

### Adding a new version

When the API contract changes enough to warrant a new published version (e.g. `2026-08-01`):

1. **Snapshot the content.** Copy `en/latest/` to `en/<version>/` (e.g. `en/2026-08-01/`). This freezes the docs for that version.
2. **Snapshot the OpenAPI spec** (if/when we adopt OpenAPI — see below). Each version should reference its own self-contained spec file. Mintlify does not support external `$ref` across files.
3. **Add a version block to `docs.json`.** Duplicate the existing `latest` block under `navigation.products[].versions`, set the new `version` string, and update all page paths from `en/latest/...` to `en/<version>/...`.
4. **Decide which version is `default: true`.** Typically the newest stable version.
5. **Tag versions** with `"tag": "Latest"` or `"tag": "Deprecated"` as appropriate — these render as badges in the version dropdown.

Example shape:

```json
"versions": [
  {
    "version": "2026-08-01",
    "default": true,
    "tag": "Latest",
    "tabs": [ ... pages under en/2026-08-01/... ]
  },
  {
    "version": "latest",
    "tag": "Deprecated",
    "tabs": [ ... pages under en/latest/... ]
  }
]
```

### Maintaining versions

- Bug fixes and clarifications that apply to all live versions need to be made in each version's directory. There is no shared-content mechanism.
- Versions that are no longer supported can be removed by deleting their `version` block from `docs.json` and the corresponding `en/<version>/` directory.

## OpenAPI and examples (future)

The Developer Portal is currently authored as MDX. When we move endpoints to OpenAPI-driven generation:

- Each version gets its own spec file under `openapi/<version>.yaml`.
- Reference it from the relevant tab in `docs.json`: `"openapi": "openapi/<version>.yaml"`.
- Request/response examples come from the spec's `examples` field.
- Per-language code samples come from `x-codeSamples` on each operation.
- Snippets can be autogenerated from the schema via `api.examples.autogenerate` in `docs.json`.

GraphQL schemas are **not** natively supported by Mintlify — GraphQL reference docs must be hand-authored in MDX.

## Common tasks

- **Add a page**: create the `.mdx` file under `en/latest/...` and add its path to the appropriate `pages` array in `docs.json`.
- **Reorder nav**: edit the `pages` / `groups` arrays in `docs.json`. Order in the file is the order users see.
- **Add an image**: drop it under `images/` and reference it with an absolute path: `/images/foo.png`.
- **Check for broken links**: `mint broken-links`.
